# The Project : University Planner
This project is meant to facilitate the planning of a bachelor degree from any university that can have its requirements translated into a form the project accepts. The "requirements" are essentially all the relationships between the objects that relate to obtaining a degree from a university. This could be a prerequisite relationship between courses. It could also be a requirement relationship where to satisfy some part of a degree you must take a group of courses before you can starting taking course from another group. One example of this could be "Lower Division" courses and "Higher Division" Courses. in order to take courses from the Higher Division you must first complete all course from the lower division.  One these relationships are given to the application the application can begin facilitating the planning process. The word "facilitate" is used because the application will handle the tedious tasks related to planning a degree. The user still has to do the work to determine what order they want to take courses in and if that is the most effecient path for them. That is not to say that theses features cannot be developed for this application. In the current state this application can only facilitate the planning process. The project however does accomplish the main task of allowing a user to plan a degree for a given university.

## Explanation of a basic degree graph
The "root" of the graph is the degree object (In the picture this a the circle with a D). The degree object is connected to requirement nodes that need to be "satisfied" by the student who wants to complete a degree. Satisfied in this context means that the student planned their courses in a way where the requirement has reached a state where all its requirements are complete. Requirement nodes are connected to course where an edge means that the course needs to be completed for the requirement to be satisfied. Courses themselves are connected to other course to denote a prerequisite relationship. Meaning that in order to take course 1 you must take course 2 first.

In this implementation of a course scheduler there are three basic operations: next, add and remove. Next returns a list of courses that can be added to a students schedule. Add and remove, as the name implies, adds and removes a course to a specific semester in a plan respectively. If you wanted to perform the next operation on this graph you would iterate through each child requirement node and look at each course and the course where all prerequisites are completed would be returned by the operation. In this case C2 would be retured as it doues not depend on any other course. Now once c2 is added to the students plan the next operation would return c1 and c3 as those course have their prerequisite completed. This process continues until all requirement nodes have all their child courses added to a users plan. At that point the degree is complete and the students ends up with a completed course plan where the student has the courses they should take in each semester.

However this situation is not present for all degrees. and moreover universities often add requireemts that allow students to take multiple paths that allow them to take a specific course. Thus a scheduler implemented using the above data structure would not be useful for many degrees. For example a course can have the option to complete one course or another. Futhermore requirements can have the same situtaion where they require that you either complete one course or the other. To add onto that, if a student want to plan a more complicated situation, like a double major, they would not be able to as this data strucutre doesnt allow for have a method to handle complicated situations.

## The actual graph data structure
The actual graph data structure adds different nodes in between the relationships that are above. And each one of those nodes has the value "OR" or "AND" associated with them. The value essentially designates  where all of the child requirements/prerequisite need to be completed in order for the requirement/prerequisite to be completed. Additionally a degree can have a relationship with another degree to indicate that for one degree to be completed that all the requirements of another must be completed.

Using the data structure details above users can describe a degree from any university and then use the application to facilitate the planning process for them.

# Technologies Used

The spring framework was used to implement the backend of the application as it comes with solutions to many of the common tasks needed when developing an application. Notably spring boot was not used and instead the Spring framework and Spring Security project were used to build the application. This was done mainly for learning purposes as it forced me to learn what Spring Boot handled for me.

For data access the hibernate framework was used in addition to a postgres database.

# Organization
Organization wise config files and entity objects are separated into their own package; Class that end in "Endpoint" contain all the request endpoint with respect to a certain aspect of the application. Within those classes there is no seperate call that handles the business logic instead I handled the business logic within that controller method. This choice was made since the application was not complex enough to warrant the use of a service class and I felt that teh use of a service class would only made the application more complex. There are however the classes CourseAddition, CourseRemoval, and CourseStateFunction which expose one function. These are the Add, Remove, and Next that were mentioned above. They were moved to seperate classes as leaving them in the EndPoint classes would be confusing.

# EndPoint Response
In general endpoints respond with the changes that occurred after action that the user requested is completed. For exmaple if a course is added the endpoint would respond with the courses that were affected to the addition of that original course.
